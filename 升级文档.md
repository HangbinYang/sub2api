# Sub2API 升级文档

## 目标和固定约束

本项目 fork 自 `Wei-Shaw/sub2api`，本地有三个需要长期保留的定制：

1. 繁体中文（以台湾用语为主）支持，包括语言推断、语言切换、日期区域和 `zh-Hant` 语言包。
2. 分组选项和标签显示用户当前的真实/有效倍率：存在用户专属倍率时显示专属倍率，否则显示分组默认倍率。
3. 管理员模式的“使用记录 → 用户排行”在用户列后显示用户备注和当前余额。

升级时必须先更新 `main`，再合并到 `codex/dev`，不得用上游文件直接覆盖这三类定制。

### 管理员用户排行定制

- 目标页面是管理员路由 `/admin/usage` 的“用户排行”Tab，不是普通用户自己的使用记录页面。
- 用户列后固定显示“备注”和“余额”两列。
- 备注取用户当前的 `notes`；余额取用户当前的 `balance`，不包含 `frozen_balance`。
- 排行中的请求数、Token 和费用按所选时间范围聚合；备注和余额是接口请求时的当前值，不是历史快照。
- 该数据只能由管理员接口返回，不得把管理员备注加入普通用户接口。

## 仓库和分支

- 上游仓库：`upstream = https://github.com/Wei-Shaw/sub2api.git`
- 自己的 fork：`origin = git@github.com:HangbinYang/sub2api.git`
- `main`：与上游最新 `main` 保持一致。
- `codex/dev`：生产使用的定制分支，包含繁体中文、真实倍率和管理员用户排行扩展。

升级前先确认工作树干净，不要覆盖用户未提交的修改。

```bash
git status --short --branch
git fetch --prune upstream
git fetch --prune origin
```

## 标准升级流程

### 1. 更新 main

`main` 只做快进合并，然后推送到 fork：

```bash
git switch main
git merge --ff-only upstream/main
git push origin main
```

### 2. 合并到 codex/dev

```bash
git switch codex/dev
git merge main
```

如有冲突，重点检查以下文件：

- `frontend/src/i18n/index.ts`
- `frontend/src/i18n/localeHelpers.ts`
- `frontend/src/i18n/locales/zh-Hant.ts`
- `frontend/src/i18n/locales/zhHantTransform.ts`
- `frontend/src/components/common/LocaleSwitcher.vue`
- `frontend/src/components/common/DateRangePicker.vue`
- `frontend/src/components/common/GroupBadge.vue`
- `frontend/src/components/common/GroupOptionItem.vue`
- `frontend/src/views/KeyUsageView.vue`
- `frontend/src/components/admin/usage/UserTokenRanking.vue`
- `frontend/src/components/admin/usage/__tests__/UserTokenRanking.spec.ts`
- `frontend/src/i18n/locales/en/admin/resources.ts`
- `frontend/src/i18n/locales/zh/admin/resources.ts`

- `backend/internal/pkg/usagestats/usage_log_types.go`

- `backend/internal/repository/usage_log_repo_breakdown_test.go`

合并后可用以下命令确认定制仍然是 `codex/dev` 相对 `main` 的独立差异：

```bash
git diff --name-status main...codex/dev
git diff main...codex/dev -- \
  frontend/src/i18n \
  frontend/src/components/common/GroupBadge.vue \
  frontend/src/components/common/GroupOptionItem.vue \
  frontend/src/components/admin/usage/UserTokenRanking.vue \
  backend/internal/pkg/usagestats/usage_log_types.go \
  backend/internal/repository/usage_log_repo_trend.go
git diff --check
```

### 3. 本地验证

```bash
cd frontend
pnpm test:run
pnpm typecheck
pnpm build
cd ..
git status --short --branch
git push origin codex/dev
```

至少要确认繁体语言推断、`zh-Hant` 语言包、倍率组件，以及管理员用户排行中的备注和当前余额没有回退。

## fancone 生产服务器

通过以下命令登录：

```bash
ssh fancone
```

服务器使用 1Panel 管理 Docker，但这两套 Sub2API 实际由下列 Compose 项目管理：

| 实例 | 容器 | Compose 目录/文件 | 宿主机端口 |
| --- | --- | --- | --- |
| 台湾实例 | `sub2api-tw` | `/home/ubuntu/sub2api-tw/docker-compose.yml` | `7484` |
| 主实例 | `sub2api` | `/home/ubuntu/sub2api/deploy/docker-compose.local.yml` + `docker-compose.codex.yml` | `7483` |

两个应用容器共用本地镜像标签 `sub2api-local:codex-dev`，代码仓库位于 `/home/ubuntu/sub2api`。镜像只需构建一次，但容器必须按顺序分别重建。

### 4. 服务器更新代码和构建镜像

```bash
git -C /home/ubuntu/sub2api status --short --branch
git -C /home/ubuntu/sub2api pull --ff-only origin codex/dev

cd /home/ubuntu/sub2api/deploy
sudo docker compose \
  -f docker-compose.local.yml \
  -f docker-compose.codex.yml \
  build sub2api
```

构建镜像不会立即改变正在运行的容器。

### 5. 先升级 sub2api-tw

升级前创建临时数据库备份，命令不会输出数据库密码：

```bash
sudo docker exec sub2api-tw-postgres sh -lc \
  'pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB"' \
  | gzip -1 > /tmp/sub2api-tw-upgrade.sql.gz
test -s /tmp/sub2api-tw-upgrade.sql.gz
```

只重建应用容器，不重启 PostgreSQL 和 Redis：

```bash
cd /home/ubuntu/sub2api-tw
sudo docker compose up -d --no-deps --force-recreate sub2api
sudo docker inspect sub2api-tw --format \
  '{{.State.Status}} {{.State.Health.Status}} {{.Image}}'
curl -fsS http://127.0.0.1:7484/health
sudo docker logs --since 10m sub2api-tw
```

只有在 `sub2api-tw` 进入 `healthy`、`/health` 返回 `{"status":"ok"}`，且日志没有 panic/fatal/迁移失败时，才能继续升级主实例。

### 6. 再升级 sub2api

```bash
cd /home/ubuntu/sub2api/deploy

main_db=$(sudo docker compose \
  -f docker-compose.local.yml \
  -f docker-compose.codex.yml \
  ps -q postgres)

sudo docker exec "$main_db" sh -lc \
  'pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB"' \
  | gzip -1 > /tmp/sub2api-upgrade.sql.gz
test -s /tmp/sub2api-upgrade.sql.gz

sudo docker compose \
  -f docker-compose.local.yml \
  -f docker-compose.codex.yml \
  up -d --no-deps --force-recreate sub2api

sudo docker inspect sub2api --format \
  '{{.State.Status}} {{.State.Health.Status}} {{.Image}}'
curl -fsS http://127.0.0.1:7483/health
sudo docker logs --since 10m sub2api
```

注意：主实例对外映射端口是 `7483`，不是容器内部的 `8080`。

### 7. 最终检查和清理

确认两个容器均为 `healthy`、使用同一个新镜像，并观察到真实 API 请求成功后，删除本次临时备份：

```bash
rm -f /tmp/sub2api-tw-upgrade.sql.gz /tmp/sub2api-upgrade.sql.gz
sudo docker image prune -f
```

不要删除应用自身的定时备份或数据卷。不要使用 `docker system prune --volumes`。
